## 2. 데이터 모델

### PostgreSQL 테이블

**diaries (일기)**
```sql
CREATE TABLE diaries (
    id              SERIAL PRIMARY KEY,
    user_id         VARCHAR(100) NOT NULL,
    diary_date      DATE NOT NULL UNIQUE,
    content         TEXT NOT NULL,           -- 생성된 일기 본문
    primary_emotion VARCHAR(20),             -- 주요 감정
    secondary_emotion VARCHAR(20),           -- 보조 감정
    emotion_intensity INTEGER,               -- 감정 강도 (1~10)
    emotion_keywords  TEXT[],                -- 감정 키워드 배열
    emotion_summary   VARCHAR(200),          -- 감정 요약
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);
```

**conversations (대화 내역)**
```sql
CREATE TABLE conversations (
    id          SERIAL PRIMARY KEY,
    diary_id    INTEGER REFERENCES diaries(id) ON DELETE CASCADE,
    role        VARCHAR(20) NOT NULL,    -- 'assistant' or 'user'
    message     TEXT NOT NULL,
    seq_order   INTEGER NOT NULL,        -- 대화 순서
    created_at  TIMESTAMP DEFAULT NOW()
);
```

**weekly_reports (주간 리포트) — Phase 2**
```sql
CREATE TABLE weekly_reports (
    id              SERIAL PRIMARY KEY,
    user_id         VARCHAR(100) NOT NULL,
    week_start      DATE NOT NULL,
    week_end        DATE NOT NULL,
    report_content  TEXT NOT NULL,
    emotion_stats   JSONB,               -- 감정 분포 통계
    created_at      TIMESTAMP DEFAULT NOW()
);
```

## 3. 핵심 데이터 흐름

### 흐름 1: 대화형 일기 작성

```
사용자: 일기 세션 시작 요청
    │
    ▼
[ChatService]
    │── 최근 3일 일기 조회 (DiaryService → PostgreSQL)
    │── 시스템 프롬프트 + 최근 일기 컨텍스트 구성
    │── Gemini API 호출 → AI 첫 질문 생성
    │
    ▼
사용자: 답변 입력
    │
    ▼
[ChatService]
    │── 대화 이력 + 새 답변으로 Gemini API 호출
    │── AI 후속 질문 생성
    │── (3~5회 반복)
    │
    ▼
[ChatService] 대화 종료 판단
    │
    ▼
[DiaryService]
    │── 전체 대화 내용으로 Gemini API 호출 → 일기 생성
    │── PostgreSQL 저장 (diaries + conversations)
    │
    ▼
[EmotionService]
    │── 일기 텍스트로 Gemini API 호출 → 감정 분석
    │── PostgreSQL 감정 필드 업데이트
    │
    ▼
[RAGService]
    │── 일기 텍스트 임베딩 → ChromaDB 저장
    │
    ▼
완료 응답 (일기 + 감정 태그)
```

### 흐름 2: 과거 일기 검색 (Phase 2)

```
사용자: "회사에서 힘들었던 날 알려줘"
    │
    ▼
[RAGService]
    │── 질문 의도 분류 (시간/감정/주제/복합)
    │── ChromaDB 시맨틱 검색 → 관련 일기 Top-K
    │── (필요시) PostgreSQL 보조 쿼리
    │── 검색 결과 + 질문으로 Gemini API 호출
    │
    ▼
응답: 관련 일기 요약 + 답변
```

### 흐름 3: 인사이트 리포트 (Phase 2)

```
사용자: "이번 주 리포트 보여줘" 또는 자동 트리거 (일요일)
    │
    ▼
[InsightService]
    │── 해당 기간 일기 조회 (PostgreSQL)
    │── 감정 통계 집계 (PostgreSQL)
    │── 전체 일기 + 통계로 Gemini API 호출
    │── 리포트 생성
    │── PostgreSQL 저장 (weekly_reports)
    │
    ▼
응답: 주간 리포트
```

## 4. 프롬프트 설계

### 대화 유도 프롬프트 (ChatService)

```
[시스템 프롬프트]
너는 사용자의 하루를 들어주는 일기 도우미야.
다음 규칙을 따라:

1. 질문은 짧고 자연스럽게 (친구처럼)
2. 한 번에 하나만 질문
3. 사용자 답변에서 감정/사건을 포착해서 후속 질문
4. 판단하거나 조언하지 말고 그냥 들어줘
5. 3~5번 대화 후 자연스럽게 마무리
6. 마무리할 때 "오늘 일기 정리해줄까?" 물어보기

[최근 일기 컨텍스트 — 있을 경우]
어제 일기: {yesterday_diary}
그저께 일기: {day_before_diary}
→ 이전 내용과 연결되는 질문을 자연스럽게 섞어줘
```

### 일기 생성 프롬프트 (DiaryService)

```
[시스템 프롬프트]
아래 대화 내용을 바탕으로 오늘의 일기를 작성해줘.

규칙:
1. 1인칭 시점, 일기체
2. 대화에서 나온 사실만 포함 (절대 지어내지 마)
3. 시간 순서대로 정리
4. 감정 표현은 사용자의 원래 표현 유지
5. 200~500자 내외
6. 날짜는 포함하지 마 (별도 저장됨)

[대화 내용]
{conversation_history}
```

### 감정 분석 프롬프트 (EmotionService)

```
[시스템 프롬프트]
아래 일기를 읽고 감정을 분석해줘.

반드시 다음 JSON 형식으로만 응답:
{
  "primary_emotion": "joy|sadness|anger|anxiety|calm|excited",
  "secondary_emotion": "joy|sadness|anger|anxiety|calm|excited|null",
  "intensity": 1~10,
  "keywords": ["키워드1", "키워드2", "키워드3"],
  "summary": "20자 이내 감정 요약"
}

[일기 내용]
{diary_content}
```

## 5. 임베딩 전략

**임베딩 모델:** gemini-embedding-001

**임베딩 텍스트 구성:**
```
일기 본문 전체
+ "감정: {primary_emotion}"
+ "키워드: {keywords}"
```

**청킹:** 일기 1건 = 1 document (일기 길이가 500자 내외로 짧아서 청킹 불필요)

**검색 설정:**
- Top-K: 5 (기본)
- 메타데이터 필터: 날짜 범위, 감정 태그

## 6. MVP 기술 결정

| 항목 | 선택 | 이유 |
|------|------|------|
| DB | PostgreSQL | pgvector 확장 가능성, JSON 지원 |
| ORM | SQLAlchemy | FastAPI 생태계 표준 |
| Vector DB | ChromaDB | 로컬 실행, 빠른 프로토타이핑 |
| LLM | Gemini 2.5 Flash | 무료 티어, 빠른 응답 |
| 인증 | 단일 사용자 (MVP) | Phase 2에서 JWT 추가 |
